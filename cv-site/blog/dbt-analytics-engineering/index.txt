d:T101d,<h2>El Antes y Después de dbt</h2>
<p>Antes de dbt, los pipelines de transformación eran un <strong>desastre</strong>:</p>
<ul>
<li>SQL scripts en Jupyter notebooks</li>
<li>Copy-paste de queries</li>
<li>Zero testing</li>
<li>Deploy manual con copy-paste a producción</li>
</ul>
<p>dbt cambió todo.</p>
<h2>¿Qué es dbt?</h2>
<p><strong>Data Build Tool</strong>: Framework para transformaciones SQL con:</p>
<ul>
<li>Versionado (Git)</li>
<li>Testing automático</li>
<li>Documentación auto-generada</li>
<li>Lineage de datos</li>
<li>CI/CD integration</li>
</ul>
<h2>Tu Primer Modelo dbt</h2>
<pre><code class="language-sql">-- models/staging/stg_transactions.sql
{{
  config(
    materialized='view'
  )
}}

SELECT
    transaction_id,
    user_id,
    CAST(amount AS FLOAT64) as amount,
    DATE(created_at) as transaction_date,
    country_code
FROM {{ source('raw', 'transactions') }}
WHERE amount > 0
</code></pre>
<h2>Testing Built-in</h2>
<pre><code class="language-yaml"># models/staging/schema.yml
version: 2

models:
  - name: stg_transactions
    description: "Transactions staging table"
    columns:
      - name: transaction_id
        tests:
          - unique
          - not_null
      - name: amount
        tests:
          - not_null
          - dbt_utils.accepted_range:
              min_value: 0
              inclusive: false
</code></pre>
<p>Correr tests:</p>
<pre><code class="language-bash">dbt test --select stg_transactions
</code></pre>
<h2>Documentación Automática</h2>
<pre><code class="language-yaml">models:
  - name: stg_transactions
    description: "Cleaned and validated transactions"
    columns:
      - name: transaction_id
        description: "Unique transaction identifier"
      - name: user_id
        description: "Reference to users table"
</code></pre>
<p>Generar docs:</p>
<pre><code class="language-bash">dbt docs generate
dbt docs serve  # Localhost:8080
</code></pre>
<p>Resultado: <strong>Documentación visual con lineage graph</strong>.</p>
<h2>Incremental Models (Para Tablas Grandes)</h2>
<pre><code class="language-sql">{{
  config(
    materialized='incremental',
    unique_key='transaction_id'
  )
}}

SELECT * FROM {{ source('raw', 'transactions') }}

{% if is_incremental() %}
  WHERE created_at > (SELECT MAX(created_at) FROM {{ this }})
{% endif %}
</code></pre>
<p>Solo procesa <strong>datos nuevos</strong> en cada corrida.</p>
<h2>Macros: DRY en SQL</h2>
<pre><code class="language-sql">-- macros/calculate_margin.sql
{% macro calculate_margin(revenue, cost) %}
  ROUND((({{ revenue }} - {{ cost }}) / {{ revenue }}) * 100, 2)
{% endmacro %}

-- Usar en modelos
SELECT
  product_id,
  {{ calculate_margin('revenue', 'cost') }} as margin_percent
FROM products
</code></pre>
<h2>Deployment con GitHub Actions</h2>
<pre><code class="language-yaml"># .github/workflows/dbt.yml
name: dbt CI/CD

on:
  push:
    branches: [main]

jobs:
  dbt-run:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install dbt
        run: pip install dbt-bigquery
      - name: Run dbt
        run: |
          dbt deps
          dbt test
          dbt run --target prod
</code></pre>
<h2>Por Qué Adoptamos dbt en MercadoLibre</h2>
<ol>
<li><strong>Version control</strong>: Todos los cambios en Git</li>
<li><strong>Testing</strong>: Captura errores antes de producción</li>
<li><strong>Documentación</strong>: Self-documenting codebase</li>
<li><strong>Reusabilidad</strong>: Macros eliminan duplicación</li>
<li><strong>Collaboration</strong>: Analysts y engineers usan mismo tool</li>
</ol>
<h2>Alternativas</h2>
<ul>
<li><strong>Dataform</strong> (Google, similar syntax)</li>
<li><strong>SQLMesh</strong> (Más features, más complejo)</li>
<li><strong>Plain SQL</strong> (No lo hagas en 2026)</li>
</ul>
<h2>Conclusión</h2>
<p>Si hacés analytics engineering y no usás dbt, <strong>estás perdiendo tiempo y dinero</strong>.</p>
<p>La inversión de aprender dbt se paga en semanas.</p>

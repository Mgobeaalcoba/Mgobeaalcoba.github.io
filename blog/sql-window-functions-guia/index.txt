f:T19ca,<h2>El Poder Oculto de SQL</h2>
<p>Si solo usás <code>GROUP BY</code> para agregaciones, <strong>te estás perdiendo del 50% de SQL</strong>.</p>
<p>Window functions te permiten hacer análisis que serían imposibles (o muy feos) con GROUP BY.</p>
<h2>¿Qué Son Window Functions?</h2>
<p>Operan sobre un "window" (ventana) de filas relacionadas, <strong>sin colapsar las filas</strong> como GROUP BY.</p>
<h3>GROUP BY (colapsa filas)</h3>
<pre><code class="language-sql">SELECT 
    country,
    SUM(amount) as total
FROM transactions
GROUP BY country
</code></pre>
<p>Output: 3 filas (una por país)</p>
<h3>Window Function (mantiene todas las filas)</h3>
<pre><code class="language-sql">SELECT 
    country,
    amount,
    SUM(amount) OVER (PARTITION BY country) as country_total
FROM transactions
</code></pre>
<p>Output: 100M filas (todas), pero cada fila tiene el total de su país.</p>
<h2>Caso #1: Ranking de Ventas</h2>
<p><strong>Pregunta:</strong> Top 3 productos por categoría.</p>
<pre><code class="language-sql">WITH ranked_products AS (
    SELECT 
        category,
        product_name,
        sales,
        ROW_NUMBER() OVER (
            PARTITION BY category 
            ORDER BY sales DESC
        ) as rank
    FROM products
)
SELECT * FROM ranked_products
WHERE rank &#x3C;= 3
</code></pre>
<p><strong>Sin window functions:</strong> Necesitarías subqueries horribles.</p>
<h2>Caso #2: Diferencia con Fila Anterior</h2>
<p><strong>Pregunta:</strong> Crecimiento día a día de ventas.</p>
<pre><code class="language-sql">SELECT 
    date,
    daily_sales,
    LAG(daily_sales) OVER (ORDER BY date) as prev_day_sales,
    daily_sales - LAG(daily_sales) OVER (ORDER BY date) as growth
FROM sales_by_day
</code></pre>
<h2>Caso #3: Running Total</h2>
<p><strong>Pregunta:</strong> Ventas acumuladas del mes.</p>
<pre><code class="language-sql">SELECT 
    date,
    daily_sales,
    SUM(daily_sales) OVER (
        ORDER BY date 
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) as cumulative_sales
FROM sales_by_day
WHERE date >= '2026-05-01'
</code></pre>
<h2>Caso #4: Moving Average (7 días)</h2>
<p><strong>Pregunta:</strong> Promedio móvil de 7 días para suavizar curva.</p>
<pre><code class="language-sql">SELECT 
    date,
    daily_sales,
    AVG(daily_sales) OVER (
        ORDER BY date 
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) as moving_avg_7d
FROM sales_by_day
</code></pre>
<h2>Caso #5: Percentiles</h2>
<p><strong>Pregunta:</strong> ¿Este usuario está en top 10% de gastos?</p>
<pre><code class="language-sql">SELECT 
    user_id,
    total_spent,
    PERCENT_RANK() OVER (ORDER BY total_spent) as percentile,
    CASE 
        WHEN PERCENT_RANK() OVER (ORDER BY total_spent) >= 0.9 
        THEN 'TOP_10%'
        ELSE 'REGULAR'
    END as segment
FROM user_spending
</code></pre>
<h2>Caso #6: First/Last Value en Grupo</h2>
<p><strong>Pregunta:</strong> Primera y última compra por usuario.</p>
<pre><code class="language-sql">SELECT DISTINCT
    user_id,
    FIRST_VALUE(purchase_date) OVER (
        PARTITION BY user_id 
        ORDER BY purchase_date
    ) as first_purchase,
    LAST_VALUE(purchase_date) OVER (
        PARTITION BY user_id 
        ORDER BY purchase_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) as last_purchase
FROM purchases
</code></pre>
<h2>Window Functions Disponibles</h2>
<h3>Ranking</h3>
<ul>
<li><code>ROW_NUMBER()</code>: 1, 2, 3, 4...</li>
<li><code>RANK()</code>: 1, 2, 2, 4... (puede repetir)</li>
<li><code>DENSE_RANK()</code>: 1, 2, 2, 3... (sin gaps)</li>
<li><code>NTILE(n)</code>: Divide en n buckets</li>
</ul>
<h3>Agregación</h3>
<ul>
<li><code>SUM()</code>, <code>AVG()</code>, <code>MIN()</code>, <code>MAX()</code>, <code>COUNT()</code></li>
</ul>
<h3>Navegación</h3>
<ul>
<li><code>LAG(col, offset)</code>: Fila anterior</li>
<li><code>LEAD(col, offset)</code>: Fila siguiente</li>
<li><code>FIRST_VALUE()</code>, <code>LAST_VALUE()</code></li>
</ul>
<h3>Estadística</h3>
<ul>
<li><code>PERCENT_RANK()</code>, <code>CUME_DIST()</code></li>
</ul>
<h2>Performance Tips</h2>
<h3>1. Ordena Por Columnas Indexadas</h3>
<pre><code class="language-sql">-- ✅ Rápido si date está indexado
OVER (ORDER BY date)

-- ❌ Lento
OVER (ORDER BY CONCAT(first_name, last_name))
</code></pre>
<h3>2. Limita el Window</h3>
<pre><code class="language-sql">-- ❌ Sin límite: escanea toda la tabla
OVER (ORDER BY date)

-- ✅ Con límite: solo últimas 100 filas
OVER (
    ORDER BY date 
    ROWS BETWEEN 99 PRECEDING AND CURRENT ROW
)
</code></pre>
<h3>3. Usa Vistas Materializadas</h3>
<p>Si ejecutás la misma window function múltiples veces, materializala.</p>
<h2>Errores Comunes</h2>
<h3>Error #1: Olvidar ORDER BY</h3>
<pre><code class="language-sql">-- ❌ Sin ORDER BY, el orden es indefinido
LAG(amount) OVER (PARTITION BY user_id)

-- ✅ Con ORDER BY, orden garantizado
LAG(amount) OVER (PARTITION BY user_id ORDER BY date)
</code></pre>
<h3>Error #2: No Entender ROWS vs RANGE</h3>
<pre><code class="language-sql">-- ROWS: Cuenta filas físicas
ROWS BETWEEN 3 PRECEDING AND CURRENT ROW  -- Exactamente 4 filas

-- RANGE: Cuenta por valor
RANGE BETWEEN INTERVAL 3 DAY PRECEDING AND CURRENT ROW  -- Últimos 3 días
</code></pre>
<h2>Caso Real en MercadoLibre</h2>
<p>Necesitábamos detectar <strong>usuarios con comportamiento anómalo</strong> (posible fraude).</p>
<pre><code class="language-sql">WITH user_behavior AS (
    SELECT 
        user_id,
        transaction_date,
        amount,
        AVG(amount) OVER (
            PARTITION BY user_id 
            ORDER BY transaction_date
            ROWS BETWEEN 30 PRECEDING AND 1 PRECEDING
        ) as avg_amount_last_30,
        STDDEV(amount) OVER (
            PARTITION BY user_id 
            ORDER BY transaction_date
            ROWS BETWEEN 30 PRECEDING AND 1 PRECEDING
        ) as stddev_last_30
    FROM transactions
)
SELECT *
FROM user_behavior
WHERE amount > avg_amount_last_30 + (3 * stddev_last_30)
-- Transacciones >3 std dev del promedio = anomalía
</code></pre>
<p><strong>Detectamos fraude en tiempo real</strong> con una query.</p>
<h2>Conclusión</h2>
<p>Window functions son <strong>game-changer para analytics</strong>.</p>
<p>Si no las usás, estás escribiendo SQL 10x más complejo de lo necesario.</p>
<p>Dominá estos 5 casos y vas a ser top 10% de data analysts.</p>

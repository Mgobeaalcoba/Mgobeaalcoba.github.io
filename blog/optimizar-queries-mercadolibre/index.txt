Procesaba 180 millones de filas y consumía 340 GB de datos escaneados por ejecución. Se ejecutaba 500 veces por día. Las matemáticas eran devastadoras.</p>
<p>Hoy esa misma query tarda <strong>4.8 segundos</strong> y escanea 8 GB. Acá te cuento exactamente cómo lo hicimos.</p>
<h2>El problema original</h2>
<p>La query calculaba métricas de performance de vendedores para el equipo de analytics. El SQL original era así:</p>
<pre><code class="language-sql">SELECT
  seller_id,
  COUNT(*) as total_orders,
  SUM(gmv) as total_gmv,
  AVG(delivery_time) as avg_delivery,
  COUNT(DISTINCT buyer_id) as unique_buyers
FROM orders_raw
WHERE DATE(created_at) BETWEEN '2025-01-01' AND '2025-12-31'
GROUP BY seller_id
ORDER BY total_gmv DESC
</code></pre>
<p>Simple, ¿no? El problema estaba en los detalles.</p>
<h2>Error #1: No usar particionamiento</h2>
<p><code>orders_raw</code> tenía <strong>12 años de datos</strong> — aproximadamente 15 billones de filas. La cláusula <code>WHERE DATE(created_at)</code> no usaba la partición de la tabla porque aplicaba una función sobre la columna particionada.</p>
<p><strong>Fix</strong>: Cambiar <code>DATE(created_at)</code> por <code>created_at</code>:</p>
<pre><code class="language-sql">WHERE created_at >= '2025-01-01'
  AND created_at &#x3C; '2026-01-01'
</code></pre>
<p>Esto redujo los datos escaneados de 340 GB a 120 GB de un solo movimiento. Ahorro: 65% de costos en esa sola línea.</p>
<h2>Error #2: <code>COUNT(DISTINCT)</code> sin pre-agrupación</h2>
<p>El <code>COUNT(DISTINCT buyer_id)</code> sobre 180 millones de filas es extremadamente costoso en BigQuery. BigQuery tiene que construir un mapa de hash gigante para cada vendedor.</p>
<p><strong>Fix</strong>: Pre-agregar en un subquery:</p>
<pre><code class="language-sql">WITH buyer_counts AS (
  SELECT
    seller_id,
    COUNT(DISTINCT buyer_id) as unique_buyers
  FROM orders_raw
  WHERE created_at >= '2025-01-01'
    AND created_at &#x3C; '2026-01-01'
  GROUP BY seller_id
),
order_metrics AS (
  SELECT
    seller_id,
    COUNT(*) as total_orders,
    SUM(gmv) as total_gmv,
    AVG(delivery_time) as avg_delivery
  FROM orders_raw
  WHERE created_at >= '2025-01-01'
    AND created_at &#x3C; '2026-01-01'
  GROUP BY seller_id
)
SELECT
  om.*,
  bc.unique_buyers
FROM order_metrics om
LEFT JOIN buyer_counts bc USING (seller_id)
ORDER BY total_gmv DESC
</code></pre>
<h2>Error #3: Clustering no configurado</h2>
<p>La tabla tenía particionamiento por fecha pero no tenía <strong>clustering</strong>. Recreamos la tabla con clustering:</p>
<pre><code class="language-sql">CREATE TABLE orders_raw_v2
PARTITION BY DATE(created_at)
CLUSTER BY country_id, seller_id
AS SELECT * FROM orders_raw;
</code></pre>
<p>Con clustering por <code>country_id</code>, las queries que filtraban por Argentina pasaron a escanear 40% menos datos automáticamente.</p>
<h2>Error #4: Materializar resultados intermedios</h2>
<p><strong>Fix</strong>: Crear una tabla materializada que se actualizaba una vez por día:</p>
<pre><code class="language-sql">CREATE OR REPLACE TABLE seller_daily_metrics
PARTITION BY metric_date
CLUSTER BY seller_id
AS (
  SELECT
    DATE(created_at) as metric_date,
    seller_id,
    country_id,
    COUNT(*) as total_orders,
    SUM(gmv) as total_gmv,
    AVG(delivery_time) as avg_delivery,
    COUNT(DISTINCT buyer_id) as unique_buyers
  FROM orders_raw
  WHERE created_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 400 DAY)
  GROUP BY 1, 2, 3
);
</code></pre>
<p>Las 500 corridas diarias ahora consultaban <code>seller_daily_metrics</code> (8 GB) en lugar de <code>orders_raw</code> (340 GB). El ahorro fue del orden de los <strong>$15,000 USD mensuales</strong> solo en esa tabla.</p>
<h2>Error #5: Proyección excesiva</h2>
<p>Varios pipelines hacían <code>SELECT *</code> cuando solo necesitaban 5-6 columnas de una tabla de 40 columnas.</p>
<p><strong>Fix</strong>: Listar explícitamente las columnas necesarias.</p>

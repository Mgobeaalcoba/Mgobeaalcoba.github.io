10:T216c,<p>En 2026, si tenés que elegir entre Airflow y Prefect para orquestar tus pipelines de datos, la decisión no es obvia. Los dos son maduros, los dos tienen comunidad activa, y los dos pueden hacer lo que necesitás. La diferencia está en el fit con tu equipo y tu escenario.</p>
<p>Usé ambas herramientas en proyectos reales. Este artículo es una comparativa honesta, sin sponsors de ningún lado.</p>
<h2>El contexto importa: antes de comparar</h2>
<p>Ninguna herramienta gana en abstracto. La que conviene depende de:</p>
<ul>
<li><strong>Tamaño del equipo</strong>: ¿Data engineers full-time o científicos que también orquestan?</li>
<li><strong>Legado</strong>: ¿Migrás desde otra herramienta o empezás desde cero?</li>
<li><strong>Cloud</strong>: ¿Usás managed services o instalación propia?</li>
<li><strong>Complejidad</strong>: Pipelines simples ETL vs workflows complejos con branching dinámico</li>
</ul>
<h2>Apache Airflow: el elefante en la habitación</h2>
<p>Airflow existe desde 2014 (Airbnb) y es, por mucho, el orquestador más usado en Data Engineering. Eso tiene ventajas y desventajas.</p>
<h3>Lo que Airflow hace bien</h3>
<p><strong>1. Madurez y ecosistema</strong>: Tiene providers para absolutamente todo — BigQuery, Snowflake, Spark, dbt, S3, Kafka, bases de datos, APIs. Si existe en el mundo de datos, hay un Airflow operator.</p>
<p><strong>2. UI madura</strong>: El DAG view, el Gantt chart, los logs inline — son funcionales y el equipo ya sabe usarlos.</p>
<p><strong>3. Managed options</strong>: Google Cloud Composer, AWS MWAA, Astronomer. Podés tener Airflow sin operarlo vos.</p>
<p><strong>4. Conocimiento del mercado</strong>: El 70% de los data engineers que contratás saben Airflow. No hay curva de aprendizaje de la herramienta.</p>
<h3>Lo que Airflow tiene difícil</h3>
<p><strong>1. DAGs son estáticos</strong>: Los DAGs se definen en tiempo de parse, no en tiempo de ejecución. Para branching dinámico basado en resultados de pasos anteriores, se pone complicado.</p>
<p><strong>2. Testing es un dolor</strong>: Testear un DAG de Airflow requiere mockear muchas cosas del internals. No es imposible, pero no es el flujo natural.</p>
<p><strong>3. Infraestructura</strong>: La instalación propia requiere Celery o Kubernetes como executor, Redis o RabbitMQ como broker, y monitoreo de todo eso. Es carga operacional real.</p>
<p><strong>4. Python mezcla con infraestructura</strong>: Los DAGs mezclan lógica de negocio con configuración de scheduling, que puede volverse un lío si no tenés estructura.</p>
<pre><code class="language-python"># DAG de Airflow — estructurado pero verbose
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.providers.google.cloud.operators.bigquery import BigQueryInsertJobOperator
from datetime import datetime, timedelta

with DAG(
    dag_id='pipeline_ventas',
    start_date=datetime(2026, 1, 1),
    schedule_interval='@daily',
    default_args={
        'retries': 3,
        'retry_delay': timedelta(minutes=5),
        'email_on_failure': True,
    },
    catchup=False,
) as dag:
    
    extraer = PythonOperator(
        task_id='extraer_datos',
        python_callable=extraer_ventas_func,
    )
    
    transformar = BigQueryInsertJobOperator(
        task_id='transformar_datos',
        configuration={
            "query": {
                "query": "{% include 'sql/transformar_ventas.sql' %}",
                "useLegacySql": False,
            }
        }
    )
    
    cargar = PythonOperator(
        task_id='cargar_resultados',
        python_callable=cargar_resultados_func,
    )
    
    extraer >> transformar >> cargar
</code></pre>
<h2>Prefect: el challenger moderno</h2>
<p>Prefect (y su versión 2.x, Prefect Orion) nació como respuesta a las limitaciones de Airflow. La filosofía es diferente: <strong>flows son código Python puro</strong> con decoradores.</p>
<h3>Lo que Prefect hace bien</h3>
<p><strong>1. Sintaxis nativa Python</strong>: Las flows de Prefect son funciones Python con <code>@flow</code> y <code>@task</code>. No hay un DSL especial.</p>
<pre><code class="language-python"># Flow de Prefect — se ve como Python normal
from prefect import flow, task
from prefect.tasks import task_input_hash
from datetime import timedelta

@task(cache_key_fn=task_input_hash, cache_expiration=timedelta(hours=1))
def extraer_ventas(fecha: str) -> list[dict]:
    return db.query(f"SELECT * FROM ventas WHERE fecha = '{fecha}'")

@task
def transformar(datos: list[dict]) -> list[dict]:
    return [calcular_margen(r) for r in datos]

@task(retries=3, retry_delay_seconds=60)
def cargar(datos: list[dict]):
    warehouse.insert_batch(datos)

@flow(name="Pipeline Ventas")
def pipeline_ventas(fecha: str):
    raw = extraer_ventas(fecha)
    transformado = transformar(raw)
    cargar(transformado)

# Para correr localmente
if __name__ == "__main__":
    pipeline_ventas(fecha="2026-01-15")
</code></pre>
<p><strong>2. Dynamic workflows</strong>: Podés crear tasks dinámicamente basándote en el resultado de tasks anteriores.</p>
<pre><code class="language-python">@flow
def pipeline_multi_pais():
    paises = obtener_paises_activos()  # task que retorna lista
    
    # Map sobre resultados dinámicamente
    resultados = procesar_pais.map(paises)
    
    consolidar(resultados)  # espera que todos terminen
</code></pre>
<p><strong>3. Testing simple</strong>: Como es Python puro, testear es straightforward:</p>
<pre><code class="language-python">def test_transformar_calcula_margen_correctamente():
    datos = [{"precio": 100, "costo": 60}]
    resultado = transformar.fn(datos)  # .fn accede a la función sin overhead de Prefect
    assert resultado[0]["margen"] == 0.40
</code></pre>
<p><strong>4. Prefect Cloud</strong>: La UI managed de Prefect es excelente — observabilidad moderna, alertas, scheduling. Y el free tier es generoso.</p>
<h3>Lo que Prefect tiene difícil</h3>
<p><strong>1. Ecosistema más pequeño</strong>: Menos integrations built-in que Airflow. Para cosas exóticas, hacés el conector vos.</p>
<p><strong>2. Adopción del mercado</strong>: Menos data engineers conocen Prefect. Curva de aprendizaje real al incorporar gente nueva.</p>
<p><strong>3. Versionado de API</strong>: Prefect 1.x vs 2.x es casi una reescritura. Si tenés flows en 1.x, la migración no es trivial.</p>

<h2>Mi recomendación: cuándo usar cada uno</h2>
<p><strong>Airflow si:</strong></p>
<ul>
<li>Tu equipo ya lo conoce o viene de otro equipo que lo usa</li>
<li>Necesitás integraciones con tecnologías específicas que solo Airflow tiene providers</li>
<li>Usás un cloud que ofrece managed Airflow (GCP con Cloud Composer es excelente)</li>
<li>La organización es grande y querés el estándar de facto de la industria</li>
</ul>
<p><strong>Prefect si:</strong></p>
<ul>
<li>Empezás desde cero sin legado</li>
<li>Tu equipo es mayormente Python developers que no quieren aprender un DSL nuevo</li>
<li>Necesitás dynamic workflows o branching complejo</li>
<li>Querés iteration velocity rápida (deploy y testear flows es más rápido)</li>
<li>El equipo es chico y no querés operar infraestructura compleja</li>
</ul>
<h2>Lo que usamos en la práctica</h2>
<p>En MercadoLibre usamos Airflow con Cloud Composer. La razón principal: ya teníamos cientos de DAGs cuando Prefect estaba madurando, y el costo de migración no se justificaba.</p>
<p>Para proyectos greenfield más chicos, Prefect es mi recomendación default hoy.</p>
<hr>
<p>La respuesta honesta: <strong>en 2026, ambas herramientas son buenas opciones</strong>. La decisión importa mucho menos que la calidad de tus pipelines, la observabilidad que implementés, y el proceso de deploy que establezcas.</p>

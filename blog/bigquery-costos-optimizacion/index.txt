10:T25f2,<p>Cuando llegué al equipo de Data en MercadoLibre, el gasto mensual en BigQuery estaba cerca de los $180,000 USD. Dieciocho meses después, con el mismo volumen de datos y más queries, lo bajamos a $54,000. Acá están las 7 técnicas que más impacto tuvieron.</p>
<h2>Por qué BigQuery puede volverse muy caro muy rápido</h2>
<p>BigQuery cobra de dos formas:</p>
<ul>
<li><strong>Por consultas</strong>: $5 USD por TB de datos escaneados (o $0.01/GB)</li>
<li><strong>Por almacenamiento</strong>: ~$0.02/GB/mes para almacenamiento activo</li>
</ul>
<p>El problema es que el pricing por consultas incentiva muy mal comportamiento: analistas que hacen <code>SELECT *</code> sobre tablas de terabytes para responder una pregunta simple.</p>
<h2>Técnica 1: Particionamiento de tablas</h2>
<p>Esta es la técnica con mayor ROI. Una tabla particionada divide los datos en segmentos (generalmente por fecha), y BigQuery solo escanea las particiones relevantes.</p>
<p><strong>Antes:</strong></p>
<pre><code class="language-sql">-- Escanea TODA la tabla (ej: 5 años de datos = 2TB)
SELECT * FROM eventos WHERE DATE(timestamp) = '2026-01-15'
</code></pre>
<p><strong>Después — crear tabla particionada:</strong></p>
<pre><code class="language-sql">CREATE TABLE eventos_particionada
PARTITION BY DATE(timestamp)
AS SELECT * FROM eventos;

-- Ahora esta query escanea solo 1 día de datos (ej: ~1GB)
SELECT * FROM eventos_particionada WHERE DATE(timestamp) = '2026-01-15'
</code></pre>
<p><strong>Importante</strong>: Para que BigQuery use la partición, el filtro debe ser directamente sobre la columna particionada. <code>WHERE DATE(timestamp) = x</code> funciona; <code>WHERE EXTRACT(YEAR FROM timestamp) = 2026</code> no siempre.</p>
<p><strong>Impacto en nuestro caso</strong>: 65% de reducción en bytes escaneados en las tablas más consultadas.</p>
<h2>Técnica 2: Clustering</h2>
<p>El clustering organiza físicamente los datos dentro de cada partición según los valores de 1-4 columnas. Cuando queries filtran por esas columnas, BigQuery puede eliminar bloques enteros sin escanearlos.</p>
<pre><code class="language-sql">CREATE TABLE pedidos_optimizada
PARTITION BY DATE(created_at)
CLUSTER BY country_id, seller_category, status
AS SELECT * FROM pedidos;
</code></pre>
<p>Ahora una query que filtre por <code>country_id = 'ARG' AND status = 'delivered'</code> puede ignorar todos los bloques de otros países y estados.</p>
<p><strong>Regla para elegir columnas de clustering:</strong></p>
<ol>
<li>Columnas que aparecen en <code>WHERE</code> con más frecuencia</li>
<li>Columnas de alta cardinalidad relativa (no sirven columnas binarias como <code>active = true</code>)</li>
<li>Máximo 4 columnas</li>
</ol>
<p><strong>Impacto</strong>: 30-50% de reducción adicional sobre las tablas ya particionadas.</p>
<h2>Técnica 3: Materializar resultados intermedios</h2>
<p>Muchos pipelines de analytics calculan las mismas agregaciones base una y otra vez. La solución es materializar esos resultados en una tabla y consultarla en lugar del raw data.</p>
<p><strong>El pattern:</strong></p>
<pre><code class="language-sql">-- Tabla de hechos materializada (actualizar 1x/día)
CREATE OR REPLACE TABLE seller_metrics_daily AS (
  SELECT
    DATE(created_at) as metric_date,
    seller_id,
    country_id,
    COUNT(*) as orders,
    SUM(gmv) as total_gmv,
    COUNT(DISTINCT buyer_id) as unique_buyers,
    AVG(delivery_days) as avg_delivery
  FROM pedidos_raw
  WHERE created_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 13 MONTH)
  GROUP BY 1, 2, 3
);
</code></pre>
<p><strong>Ahora las queries de los analistas:</strong></p>
<pre><code class="language-sql">-- En lugar de escanear la tabla raw (200GB), escanea la materializada (2GB)
SELECT
  seller_id,
  SUM(total_gmv) as gmv_ultimo_trimestre,
  SUM(orders) as orders_ultimo_trimestre
FROM seller_metrics_daily
WHERE metric_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
GROUP BY seller_id
ORDER BY gmv_ultimo_trimestre DESC
LIMIT 100;
</code></pre>
<p><strong>Impacto</strong>: 95%+ de reducción en bytes para queries analíticas sobre tablas materializadas.</p>
<h2>Técnica 4: Proyección columnar — no uses SELECT *</h2>
<p>BigQuery es columnar: cuando hacés <code>SELECT *</code>, escanea TODAS las columnas, aunque tu análisis solo necesite 3.</p>
<p><strong>El error más común:</strong></p>
<pre><code class="language-sql">-- Tabla con 40 columnas. Solo necesitás 3. Escaneás 40.
SELECT *
FROM transacciones
WHERE fecha = '2026-01-15'
  AND country = 'ARG'
LIMIT 100
</code></pre>
<p><strong>El fix (tedioso pero importante):</strong></p>
<pre><code class="language-sql">SELECT transaction_id, amount, status
FROM transacciones
WHERE fecha = '2026-01-15'
  AND country = 'ARG'
LIMIT 100
</code></pre>
<p>En una tabla de 1TB con 40 columnas y solo necesitás 3, esto te da una reducción de <strong>92.5%</strong> de datos escaneados.</p>
<p><strong>Cómo hacerlo cumplir en equipo</strong>: Configurar alertas en GCP cuando una query escanea más de X GB. Después hay una conversación educativa.</p>
<h2>Técnica 5: Usar Reservations en lugar de on-demand</h2>
<p>Si tu empresa tiene una carga de BigQuery predecible, las <strong>BigQuery Reservations</strong> (slots comprometidos) pueden ser mucho más baratas que el pricing on-demand.</p>
<ul>
<li>On-demand: $5/TB escaneado, sin límite de concurrencia</li>
<li>Reservations: Precio fijo por capacidad de procesamiento (slots), ideal para cargas predecibles</li>
</ul>
<p><strong>Cuándo cambia el análisis</strong>: Si tu gasto on-demand supera ~$10,000 USD/mes, vale la pena analizar reservations. El breakeven depende de tu patrón de uso.</p>
<p>En nuestro caso, pasamos un 60% de nuestras cargas a reservations (el batch diario) y mantuvimos el 40% ad-hoc en on-demand. Esto redujo el costo total en 25% adicional.</p>
<h2>Técnica 6: Table expiration y archivado</h2>
<p>Datos viejos que nadie consulta siguen cobrando almacenamiento. BigQuery permite configurar expiración automática:</p>
<pre><code class="language-sql">-- Configurar expiración a 90 días para tablas temporales
CREATE TABLE logs_temporales
OPTIONS (expiration_timestamp = TIMESTAMP_ADD(CURRENT_TIMESTAMP(), INTERVAL 90 DAY))
AS SELECT * FROM logs_raw WHERE DATE(created_at) = CURRENT_DATE();
</code></pre>
<p>Para datos históricos que se consultan raramente, moverlos a <strong>long-term storage</strong> (automático después de 90 días sin modificación) reduce el costo de almacenamiento de $0.02/GB a $0.01/GB.</p>
<p>Identificar tablas candidatas:</p>
<pre><code class="language-sql">SELECT
  table_id,
  ROUND(size_bytes / POW(10, 9), 2) as size_gb,
  DATE(TIMESTAMP_MILLIS(last_modified_time)) as last_modified,
  DATE_DIFF(CURRENT_DATE(), DATE(TIMESTAMP_MILLIS(last_modified_time)), DAY) as days_since_modified
FROM `project.dataset.__TABLES__`
WHERE DATE_DIFF(CURRENT_DATE(), DATE(TIMESTAMP_MILLIS(last_modified_time)), DAY) > 180
ORDER BY size_bytes DESC;
</code></pre>
<p><strong>Impacto</strong>: 15% de reducción en costos de almacenamiento.</p>
<h2>Técnica 7: Query caching y scheduled queries</h2>
<p>BigQuery cachea resultados por 24hs. Aprovechar esto:</p>
<pre><code class="language-python"># Airflow DAG: correr la query pesada una vez y guardar
def ejecutar_y_cachear():
    client = bigquery.Client()
    
    job_config = bigquery.QueryJobConfig(
        destination=f"project.dataset.resultado_diario_{datetime.today().strftime('%Y%m%d')}",
        write_disposition=bigquery.WriteDisposition.WRITE_TRUNCATE,
        use_query_cache=True,
    )
    
    query = """
        SELECT ... FROM tabla_grande ...
    """
    
    job = client.query(query, job_config=job_config)
    job.result()
</code></pre>
<p>Con Scheduled Queries en BigQuery, podés materializar resultados automáticamente en horarios de bajo costo (madrugada) para que estén disponibles durante el día.</p>
<h2>El dashboard de costos que te recomiendo implementar</h2>
<p>Antes de optimizar, necesitás visibilidad. Este es el dataset de monitoreo que implementamos:</p>
<pre><code class="language-sql">-- ¿Quién gasta más? ¿En qué queries?
SELECT
  user_email,
  COUNT(*) as query_count,
  ROUND(SUM(total_bytes_billed) / POW(10, 12) * 5, 2) as costo_usd,
  ROUND(AVG(total_bytes_billed) / POW(10, 9), 1) as avg_gb_por_query
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND job_type = 'QUERY'
  AND state = 'DONE'
GROUP BY user_email
ORDER BY costo_usd DESC
LIMIT 20;
</code></pre>
<p>Esta query sola te dice quiénes son los "top consumers" y dónde empezar.</p>

<p>La revisión de código es una práctica fundamental en el desarrollo de software, esencial para garantizar la calidad, la seguridad y la mantenibilidad del código. Tradicionalmente, este proceso se ha realizado de forma manual, confiando en la experiencia y la diligencia de los desarrolladores para identificar errores y posibles mejoras. Sin embargo, el advenimiento de la inteligencia artificial, y en particular de los modelos de lenguaje grande (LLM) como Claude, ha abierto nuevas vías para automatizar y mejorar significativamente este proceso. A pesar de los avances, las herramientas de revisión de código basadas en IA existentes a menudo presentan limitaciones en cuanto a la profundidad del análisis, la gestión del contexto y la capacidad de abordar revisiones complejas de manera sistemática.</p>
<p>Este artículo técnico presenta <code>adamsreview</code>, una extensión para Claude Code diseñada para superar estas limitaciones. <code>adamsreview</code> introduce un enfoque de revisión de código multi-agente, que opera en varias etapas y emplea técnicas avanzadas como la paralelización de sub-agentes, pases de validación dedicados, gestión de estado persistente a través de artefactos JSON y la opción de consolidar revisiones mediante un ensamble de modelos (incluido Codex) y comentarios automatizados de bots de PR. El objetivo es proporcionar un análisis más exhaustivo y preciso que las soluciones integradas o de terceros, reduciendo al mismo tiempo los falsos positivos.</p>
<h2>Arquitectura y Componentes de adamsreview</h2>
<p><code>adamsreview</code> se concibe como un conjunto de seis comandos slash (<code>/review</code>, <code>/codex-review</code>, <code>/add</code>, <code>/promote</code>, <code>/walkthrough</code>, <code>/fix</code>) integrados en el entorno de Claude Code. Estos comandos están diseñados para interactuar de manera coherente y escalar la complejidad de la revisión de código, desde un análisis inicial hasta la corrección de errores identificados.</p>
<h3>1. Comandos Principales y Flujo de Trabajo</h3>
<ul>
<li><code>/review</code>: Inicia una revisión de código estándar, similar al comando <code>/review</code> integrado en Claude Code, pero con la capacidad subyacente de un análisis más profundo y multi-etapa.</li>
<li><code>/codex-review</code>: Permite la integración de Codex (u otros modelos de código) para una revisión complementaria o paralela, aprovechando las fortalezas específicas de diferentes modelos.</li>
<li><code>/add</code>: Añade nuevas funcionalidades o archivos a la revisión en curso, permitiendo una iteración incremental sobre la base de código.</li>
<li><code>/promote</code>: Promociona los hallazgos o sugerencias de una etapa de revisión a una etapa superior o a la consideración final.</li>
<li><code>/walkthrough</code>: Facilita una interacción guiada con el usuario para discutir hallazgos complejos o elementos que requieren juicio humano.</li>
<li><code>/fix</code>: Desencadena el proceso de corrección de los errores identificados, involucrando agentes dedicados y validaciones posteriores.</li>
</ul>
<p>El flujo de trabajo general está diseñado para ser iterativo y modular. Un comando inicial como <code>/review</code> puede desencadenar una serie de sub-agentes que analizan diferentes aspectos del código (lógica, seguridad, estilo, rendimiento). Los resultados de estas sub-revisiones se agregan y gestionan mediante el estado persistente.</p>
<h3>2. Gestión de Estado Persistente (JSON Artifacts)</h3>
<p>Una de las innovaciones clave de <code>adamsreview</code> es su mecanismo de gestión de estado persistente. A diferencia de las herramientas que dependen exclusivamente de la memoria volátil de la sesión del LLM, <code>adamsreview</code> almacena el estado de la revisión (hallazgos, comentarios, decisiones, contexto relevante) en archivos JSON en el disco.</p>
<h4>Propósito de la Gestión de Estado</h4>
<ul>
<li><strong>Persistencia del Contexto:</strong> Permite que el contexto de la revisión persista a través de interacciones separadas o reinicios de la sesión de Claude Code. Esto es crucial para revisiones de larga duración o para volver a una revisión en un momento posterior.</li>
<li><strong>Modularidad y Reutilización:</strong> El estado almacenado puede ser cargado y modificado por diferentes comandos o agentes, permitiendo un flujo de trabajo más granular. Por ejemplo, un agente de corrección puede cargar el estado de los errores identificados por un agente de revisión y luego actualizarlo con las soluciones aplicadas.</li>
<li><strong>Depuración y Auditoría:</strong> Los archivos JSON sirven como un registro detallado del proceso de revisión, facilitando la depuración de la propia herramienta y proporcionando una pista de auditoría de las decisiones tomadas durante la revisión.</li>
<li><strong>Integración Externa:</strong> El formato JSON permite una fácil integración con herramientas externas o scripts de automatización.</li>
</ul>
<h4>Implementación y Manipulación</h4>
<p>El sistema incluye scripts integrados para mantener actualizados estos artefactos JSON. Estos scripts son responsables de serializar el estado de la revisión, actualizarlo con nuevos hallazgos o modificaciones y deserializarlo para su uso por otros componentes.</p>

<h3>3. Agentes Paralelos y Pasadas de Validación</h3>
<p><code>adamsreview</code> utiliza un enfoque de agentes múltiples, donde cada agente está especializado en una tarea particular. Estos agentes pueden ejecutarse en paralelo para acelerar el proceso de revisión.</p>
<h4>Paralelización de Sub-Agentes</h4>
<p>En lugar de un único LLM que intente abarcar todos los aspectos de la revisión, <code>adamsreview</code> puede desglosar la tarea en sub-tareas asignadas a agentes específicos. Por ejemplo:</p>
<ul>
<li><strong>Agente de Lógica:</strong> Se enfoca en la corrección de errores algorítmicos, condiciones de carrera y manejo de excepciones.</li>
<li><strong>Agente de Seguridad:</strong> Busca vulnerabilidades como inyecciones SQL, XSS, manejo inseguro de credenciales.</li>
<li><strong>Agente de Estilo y Convenciones:</strong> Verifica el cumplimiento de guías de estilo (PEP 8 para Python, etc.) y convenciones del equipo.</li>
<li><strong>Agente de Rendimiento:</strong> Identifica cuellos de botella potenciales, uso ineficiente de memoria o CPU.</li>
<li><strong>Agente de Documentación:</strong> Revisa la calidad y completitud de los comentarios y la documentación.</li>
</ul>
<p>Estos agentes se ejecutan en paralelo, cada uno procesando su porción del código o su aspecto de análisis, y luego sus resultados se agregan.</p>
<h4>Pasadas de Validación</h4>
<p>La revisión no se limita a una única pasada. <code>adamsreview</code> puede implementar múltiples pasadas de validación:</p>
<ul>
<li><strong>Primera Pasada (Análisis Inicial):</strong> Ejecución de todos los agentes especializados en paralelo para identificar hallazgos iniciales.</li>
<li><strong>Pasada de Consolidación:</strong> Agregación y eliminación de duplicados de los hallazgos de la primera pasada. Los hallazgos se almacenan en el estado JSON.</li>
<li><strong>Pasada de Corrección:</strong> Si se activa el comando <code>/fix</code>, agentes específicos intentan corregir los problemas identificados.</li>
<li><strong>Pasada de Regresión:</strong> Después de aplicar las correcciones, se ejecuta una pasada de validación para asegurar que las correcciones no hayan introducido nuevos errores (regresiones) en otras partes del código. Esta pasada puede ser menos exhaustiva pero enfocada en verificar la integridad general.</li>
</ul>
<h3>4. Comando <code>/walkthrough</code> y Interacción Humana</h3>
<p>Uno de los desafíos de la automatización de revisiones de código es la interpretación de hallazgos ambiguos o la necesidad de juicios contextuales que solo un humano puede proporcionar. El comando <code>/walkthrough</code> aborda esto utilizando la función <code>AskUserQuestion</code> de Claude.</p>
<p>Cuando el comando <code>/walkthrough</code> se activa, <code>adamsreview</code> puede presentar al usuario una lista de hallazgos que requieren su atención. Para cada hallazgo, puede:</p>
<ul>
<li><strong>Describir el hallazgo:</strong> Explicar el problema potencial, las líneas de código afectadas y la razón por la que se marcó.</li>
<li><strong>Preguntar por aclaración:</strong> Si el hallazgo es ambiguo o su impacto no es claro, puede usar <code>AskUserQuestion</code> para solicitar al usuario que confirme si el problema es real o si existe una justificación para el comportamiento observado.</li>
<li><strong>Solicitar acción:</strong> Pedir al usuario que decida si el hallazgo debe ser corregido, ignorado o si necesita más información.</li>
</ul>
<p>Este proceso interactivo asegura que las revisiones se mantengan precisas y evita la generación excesiva de "ruido" o falsos positivos que obliguen al desarrollador a revisar manualmente cada punto.</p>
<h3>5. Comando <code>/fix</code> y Gestión de Correcciones</h3>
<p>El comando <code>/fix</code> está diseñado para automatizar la corrección de los errores identificados. Este proceso se realiza en varias etapas:</p>
<ol>
<li><strong>Agrupación de Correcciones:</strong> Los hallazgos se agrupan por tipo o por archivo para optimizar el proceso de corrección.</li>
<li><strong>Despliegue de Agentes de Corrección:</strong> Agentes especializados (posiblemente utilizando modelos como Claude Opus o incluso enfoques de codificación más directos) se encargan de generar las modificaciones de código necesarias para abordar cada hallazgo.</li>
<li><strong>Aplicación de Cambios (Preliminar):</strong> Las correcciones propuestas se aplican al código, pero aún no se confirman permanentemente.</li>
<li><strong>Revisión Post-Corrección:</strong> El código modificado se somete nuevamente a un análisis (potencialmente con un modelo como Opus) para verificar la eficacia de las correcciones y detectar regresiones.</li>
<li><strong>Commit de Supervivientes:</strong> Solo las correcciones que pasan la validación post-corrección se aplican permanentemente (por ejemplo, mediante un commit de borrador o la preparación de un PR). Las regresiones o correcciones fallidas se revierten o se marcan para una revisión manual.</li>
</ol>
<p>Este enfoque de "corrección y validación" ayuda a mantener la integridad del código a medida que se aplican correcciones automáticas.</p>
<h3>6. Ensamble de Modelos y Codex CLI</h3>
<p>La flexibilidad para integrar diferentes modelos de IA es una característica distintiva de <code>adamsreview</code>. Además de Claude, el comando <code>/codex-review</code> permite la ejecución de revisiones utilizando Codex CLI, aprovechando la robustez de los modelos de código de OpenAI.</p>
<h4>Ensamble de Revisión</h4>
<p>La idea del ensamble es combinar las fortalezas de múltiples modelos. Por ejemplo:</p>
<ul>
<li>Claude Opus podría ser el principal motor de revisión, con su amplia capacidad de comprensión contextual y razonamiento.</li>
<li>Codex podría usarse para identificar patrones de código específicos o para verificar la sintaxis y semántica de manera más precisa en ciertos lenguajes.</li>
</ul>
<p>Los resultados de estos diferentes modelos se fusionan, y las discrepancias o los hallazgos complementarios se reportan. Esto puede llevar a una cobertura más completa y a una mayor confianza en los hallazgos.</p>
<h4>Integración con PR Bot Comments</h4>
<p>La automatización de <code>adamsreview</code> se extiende a la comunicación. Los hallazgos y las decisiones del proceso de revisión pueden ser automáticamente publicados como comentarios en Pull Requests (PRs) a través de la integración con bots de PR. Esto mantiene a los equipos informados sobre el estado de la revisión y los problemas identificados sin necesidad de intervención manual.</p>
<h3>7. Diferencias con <code>/ultrareview</code> y Consideraciones de Costo</h3>
<p>Es importante distinguir <code>adamsreview</code> de comandos como <code>/ultrareview</code> de Claude. Mientras que <code>/ultrareview</code> puede ofrecer una revisión de alto nivel, a menudo se carga contra el "Extra Usage Pool" de la suscripción de Claude, lo que puede generar costos adicionales significativos. <code>adamsreview</code>, al ejecutarse dentro de la suscripción estándar de Claude Code (se recomienda el plan Max por su mayor cuota de tokens y capacidad de procesamiento), ofrece una alternativa más predecible en términos de costos, aprovechando la infraestructura ya pagada.</p>
<h2>Implementación Práctica y Casos de Uso</h2>
<p>La implementación de <code>adamsreview</code> se realiza a través de la instalación del plugin en Claude Code:</p>
<pre><code class="language-bash">/plugin marketplace add adamjgmiller/adamsreview
/plugin install adamsreview@adamsreview
</code></pre>
<p>Una vez instalado, los comandos pueden ser invocados directamente en la interfaz de Claude Code.</p>
<h3>Ejemplo de Flujo de Trabajo para un PR</h3>
<ol>
<li>
<p><strong>Inicio de Revisión:</strong> El desarrollador abre un Pull Request y, en la conversación de Claude Code, invoca:
<code>/adamsreview review --repo &#x3C;url_del_repo> --branch &#x3C;nombre_de_la_rama></code>
o si ya está en el contexto de un PR:
<code>/adamsreview review</code></p>
</li>
<li>
<p><strong>Análisis Inicial:</strong> <code>adamsreview</code> detecta los cambios en el PR, inicia los agentes de revisión paralelos (lógica, seguridad, estilo, etc.) y almacena los hallazgos iniciales en un archivo JSON (<code>review_state.json</code> en el directorio de trabajo).</p>
</li>
<li>
<p><strong>Revisión de Hallazgos (Opcional):</strong> Si hay hallazgos que requieren aclaración, el desarrollador puede invocar:
<code>/adamsreview walkthrough</code>
Claude interactúa con el desarrollador para discutir los puntos sensibles. Las decisiones (corregir, ignorar) se actualizan en <code>review_state.json</code>.</p>
</li>
<li>
<p><strong>Corrección de Errores:</strong> El desarrollador puede decidir automatizar la corrección de los errores marcados:
<code>/adamsreview fix</code>
Los agentes de corrección actúan, aplican cambios propuestos y se realiza una pasada de validación de regresión. Los cambios exitosos se preparan para ser commit.</p>
</li>
<li>
<p><strong>Revisión Complementaria (Opcional):</strong> Si se desea una perspectiva adicional, se puede ejecutar una revisión con otro modelo:
<code>/adamsreview codex-review</code>
Los hallazgos de Codex se fusionan con el estado existente.</p>
</li>
<li>
<p><strong>Promoción y Finalización:</strong> Los hallazgos finales y las correcciones aplicadas pueden ser "promocionados" para generar un informe de resumen o para ser automáticamente comentados en el PR:
<code>/adamsreview promote --level final</code></p>
</li>
</ol>
<h3>Beneficios Clave Observados</h3>
<ul>
<li><strong>Mayor Detección de Bugs:</strong> En las revisiones del propio autor, <code>adamsreview</code> ha identificado significativamente más errores reales en comparación con Claude Code <code>/review</code>, <code>/ultrareview</code>, CodeRabbit, Greptile y la revisión integrada de Codex.</li>
<li><strong>Reducción de Falsos Positivos:</strong> El enfoque multi-agente y las pasadas de validación contribuyen a una mayor precisión, minimizando las alertas irrelevantes.</li>
<li><strong>Profundidad de Análisis:</strong> La ejecución de múltiples agentes especializados y la gestión de estado permiten un análisis más profundo y matizado que una única pasada de un solo modelo.</li>
<li><strong>Eficiencia y Escalabilidad:</strong> La paralelización y la persistencia del estado agilizan el proceso de revisión, haciéndolo más escalable para proyectos grandes y revisiones continuas.</li>
<li><strong>Interacción Controlada:</strong> El comando <code>/walkthrough</code> permite un control humano efectivo sobre los hallazgos ambiguos, manteniendo el equilibrio entre automatización y juicio experto.</li>
</ul>
<h2>Consideraciones Técnicas y Futuras Direcciones</h2>
<p>La implementación efectiva de <code>adamsreview</code> se basa en varios pilares técnicos:</p>
<ul>
<li><strong>Orquestación de Agentes:</strong> Un componente central es el orquestador que gestiona la creación, ejecución, comunicación y agregación de resultados de los múltiples agentes.</li>
<li><strong>Serialización Eficiente:</strong> La elección de JSON como formato de estado es una decisión práctica por su ubicuidad y facilidad de parseo, pero para estados muy grandes, se podrían explorar formatos binarios o bases de datos ligeras.</li>
<li><strong>Manejo de Contexto del LLM:</strong> La clave para el éxito es cómo se pasan los fragmentos de código, los hallazgos y el estado a los modelos de IA. Esto implica estrategias de chunking, summarization y prompt engineering cuidadosas para cada agente y cada etapa.</li>
<li><strong>Gestión de Dependencias y Entorno:</strong> Asegurar que los scripts de utilidades y los comandos de plugin funcionen correctamente dentro del entorno de Claude Code es fundamental.</li>
</ul>
<p>Las futuras direcciones de desarrollo podrían incluir:</p>
<ul>
<li><strong>Integración de Modelos Adicionales:</strong> Ampliar la compatibilidad con otros LLMs o herramientas de análisis estático especializadas.</li>
<li><strong>Aprendizaje Continuo:</strong> Permitir que el sistema aprenda de las correcciones del usuario para refinar sus modelos de detección de errores y sus estrategias de corrección.</li>
<li><strong>Análisis Predictivo de Riesgos:</strong> Utilizar métricas de código y patrones de cambios para predecir áreas de alto riesgo que requieran revisiones más exhaustivas.</li>
<li><strong>Soporte para Múltiples Lenguajes:</strong> Mejorar la adaptabilidad a un espectro más amplio de lenguajes de programación.</li>
<li><strong>Interfaz Gráfica/Visualización:</strong> Desarrollar herramientas complementarias para visualizar el estado de la revisión y los hallazgos de manera más intuitiva.</li>
</ul>
<h2>Conclusión</h2>
<p><code>adamsreview</code> representa un avance significativo en la aplicación de IA para la revisión de código, yendo más allá de las soluciones existentes al implementar un marco robusto de agentes múltiples, gestión de estado persistente y flujos de trabajo interactivos. Al abordar las limitaciones de profundidad, contexto y precisión, <code>adamsreview</code> tiene el potencial de transformar la forma en que los desarrolladores abordan la calidad del código, permitiendo la detección temprana de errores y la mejora continua del software. Este enfoque no solo aumenta la eficiencia sino que también fomenta una cultura de código más sólida y segura.</p>

10:T1c66,<p>Antes de dbt, la transformación de datos en muchos equipos era un caos: queries SQL ad-hoc en notebooks, stored procedures sin versionado, pipelines en Python que mezclaban extracción, transformación y carga. Nadie sabía exactamente de dónde venían los datos que usaban los dashboards.</p>
<p>dbt (data build tool) resolvió este problema definiendo un estándar para el analytics engineering que hoy es adoptado por prácticamente toda empresa de datos seria.</p>
<h2>¿Qué es dbt y qué problema resuelve?</h2>
<p>dbt es una herramienta de línea de comandos que te permite escribir transformaciones de datos como <strong>modelos SQL</strong> versionados, testeables y documentados.</p>
<p><strong>El antes (sin dbt):</strong></p>
<pre><code>Analista escribe SQL ad-hoc → lo copia a un stored procedure → alguien lo modifica → 
nadie sabe qué cambió → el dashboard muestra números inconsistentes → reunión de 3 horas
</code></pre>
<p><strong>El después (con dbt):</strong></p>
<pre><code>Modelo SQL en git → PR review → tests automáticos → deploy → documentación generada automáticamente
</code></pre>
<p>La filosofía: los datos ya llegaron al warehouse. dbt solo hace la capa de <strong>transformación</strong> (la T de ELT). No extrae ni carga datos — eso lo hace tu pipeline (Airbyte, Fivetran, Kafka, etc.).</p>
<h2>Los conceptos clave</h2>
<h3>Modelos</h3>
<p>Un modelo en dbt es un archivo <code>.sql</code> con una SELECT statement. dbt lo ejecuta y crea la tabla o vista correspondiente en tu warehouse.</p>
<pre><code class="language-sql">-- models/marts/ventas_diarias.sql
with pedidos as (
    select * from {{ ref('stg_pedidos') }}
),

vendedores as (
    select * from {{ ref('stg_vendedores') }}
)

select
    date_trunc('day', p.created_at) as fecha,
    p.seller_id,
    v.nombre_vendedor,
    count(*) as total_pedidos,
    sum(p.monto) as gmv,
    count(distinct p.buyer_id) as compradores_unicos
from pedidos p
left join vendedores v using (seller_id)
group by 1, 2, 3
</code></pre>
<p>La macro <code>{{ ref('stg_pedidos') }}</code> es la magia: dbt resuelve las dependencias automáticamente y construye el DAG de transformaciones.</p>
<h3>Estructura de carpetas estándar</h3>
<pre><code>models/
├── staging/           # Datos crudos limpiados, renombrados
│   ├── stg_pedidos.sql
│   ├── stg_vendedores.sql
│   └── stg_productos.sql
├── intermediate/      # Lógica compleja, combinaciones
│   └── int_pedidos_enriquecidos.sql
└── marts/             # Tablas finales para consumo
    ├── fct_ventas_diarias.sql
    └── dim_vendedores.sql
</code></pre>
<p>Esta convención (staging → intermediate → marts) es el estándar de la industria para organizar transformaciones.</p>
<h3>Tests</h3>
<p>dbt incluye tests out-of-the-box:</p>
<pre><code class="language-yaml"># models/schema.yml
models:
  - name: ventas_diarias
    columns:
      - name: seller_id
        tests:
          - not_null
          - relationships:
              to: ref('dim_vendedores')
              field: seller_id
      - name: fecha
        tests:
          - not_null
      - name: gmv
        tests:
          - not_null
          - dbt_utils.expression_is_true:
              expression: ">= 0"
</code></pre>
<p>También podés escribir tests singulares (queries SQL que no deben retornar filas):</p>
<pre><code class="language-sql">-- tests/ventas_no_tienen_gmv_negativo.sql
select *
from {{ ref('ventas_diarias') }}
where gmv &#x3C; 0
</code></pre>
<p>Si la query retorna filas, el test falla. Simple y poderoso.</p>
<h3>Documentación automática</h3>
<pre><code class="language-yaml">models:
  - name: ventas_diarias
    description: "Métricas diarias de ventas por vendedor"
    columns:
      - name: gmv
        description: "Gross Merchandise Value — valor total de pedidos, sin devoluciones"
      - name: compradores_unicos
        description: "Cantidad de compradores únicos (distinct buyer_id) en el día"
</code></pre>
<p><code>dbt docs generate</code> crea un sitio web con el linaje de datos completo. Nunca más "¿de dónde sale este número?".</p>
<h2>Flujo de trabajo con dbt</h2>
<pre><code class="language-bash"># Iniciar un proyecto nuevo
dbt init mi_proyecto

# Correr todos los modelos
dbt run

# Correr solo modelos específicos
dbt run --select ventas_diarias
dbt run --select +ventas_diarias  # con todos sus ancestros

# Correr tests
dbt test

# Generar y servir documentación
dbt docs generate
dbt docs serve

# Run + test en un comando
dbt build
</code></pre>
<h2>Macros: SQL DRY</h2>
<p>dbt usa Jinja templating para macros reutilizables:</p>
<pre><code class="language-sql">-- macros/fecha_argentina.sql
{% macro fecha_argentina(columna) %}
    CONVERT_TIMEZONE('UTC', 'America/Argentina/Buenos_Aires', {{ columna }})
{% endmacro %}
</code></pre>
<pre><code class="language-sql">-- Uso en cualquier modelo
select
    {{ fecha_argentina('created_at') }} as created_at_arg,
    ...
from pedidos
</code></pre>
<h2>Materialización: cuándo usar views vs tables vs incremental</h2>
<p><strong>View</strong>: Para modelos ligeros que se calculan al momento de consulta. Rápido de actualizar, caro de consultar si es complejo.</p>
<p><strong>Table</strong>: Se recalcula completo en cada <code>dbt run</code>. Bueno para modelos finales que se consultan frecuentemente.</p>
<p><strong>Incremental</strong>: Solo procesa datos nuevos desde la última corrida. Esencial para tablas grandes:</p>
<pre><code class="language-sql">-- models/fct_eventos.sql
{{ config(materialized='incremental', unique_key='event_id') }}

select
    event_id,
    user_id,
    event_type,
    created_at
from source_events

{% if is_incremental() %}
  -- Solo procesa eventos después del último registro en la tabla
  where created_at > (select max(created_at) from {{ this }})
{% endif %}
</code></pre>
<h2>dbt Cloud vs dbt Core</h2>
<ul>
<li><strong>dbt Core</strong>: Open source, corré en tu máquina o en tu CI/CD</li>
<li><strong>dbt Cloud</strong>: SaaS con IDE, scheduler, CI/CD integrado, alertas</li>
</ul>
<p>Para equipos pequeños, dbt Core + GitHub Actions + Airflow/Prefect para scheduling es suficiente y gratis. Para equipos medianos/grandes, dbt Cloud hace la operación mucho más simple.</p>
<h2>El impacto real</h2>
<p>En los proyectos donde implementé dbt, los cambios más notables fueron:</p>
<ol>
<li><strong>Tiempo de respuesta a preguntas de negocio</strong>: De días a horas, porque los datos ya están documentados y el linaje es visible</li>
<li><strong>Confianza en los datos</strong>: Los tests automáticos atrapan regresiones antes de que lleguen a producción</li>
<li><strong>Onboarding de nuevos analistas</strong>: En vez de semanas entendiendo el sistema, con <code>dbt docs serve</code> tienen el mapa completo desde el día 1</li>
<li><strong>Code review de transformaciones</strong>: Por primera vez, las transformaciones de datos pasaban por PR review como código de aplicación</li>
</ol>
<hr>
<p>Si tu equipo todavía hace transformaciones SQL sin versionado ni tests, dbt es la mejora de mayor ROI que podés implementar este trimestre. El setup inicial lleva un día; el retorno se ve en semanas.</p>
